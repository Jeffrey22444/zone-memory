# Zone Memory

Use this skill to set up a lean project operating system:

- Planning / Execution / Acceptance / Maintenance zones
- `docs/project_notes/` memory files
- optional tracked `docs/product_consensus/...`, `docs/architecture.md`, and `docs/implementation-plan.md`
- `AGENTS.md` and optional `CLAUDE.md` protocols
- Chinese zone naming and opening prompts by default
- automatic Git bootstrap for the current workspace
- automatic creation of the four zone threads
- minimal first replies from freshly created zone threads
- fixed model and thinking defaults per zone
- short handoffs and low-token review flow

## Quick Start

1. Inspect existing `AGENTS.md`, `CLAUDE.md`, `docs/`, and project conventions.
2. Create or update:
   - `docs/project_notes/bugs.md`
   - `docs/project_notes/decisions.md`
   - `docs/project_notes/key_facts.md`
   - `docs/project_notes/issues.md`
   - `docs/project_notes/zone_operating_model.md`
3. For greenfield or under-specified projects, also create or update:
   - `docs/product_consensus/...`
   - `docs/architecture.md`
   - `docs/implementation-plan.md`
4. Add or merge the workflow section from [references/agents_section.md](references/agents_section.md).
5. Seed the project-note files from `references/`.
6. Initialize Git for the current workspace if it is not already a repository.
7. Create the four zones automatically and send each one an opening prompt.
8. Log the bootstrap in `docs/project_notes/issues.md`.

## Defaults

- Keep stable rules in `AGENTS.md` and `zone_operating_model.md`, not in every prompt.
- Task cards should default to a minimum required read set; route unchanged product, architecture, or phase docs as on-demand reads instead of mandatory rereads.
- Keep `issues.md` two-layered: current summary first, chronological log below.
- Keep `key_facts.md` stable-facts-first; drift-prone items go in a short "Last Verified Environment" section.
- Every execution task gets a short Acceptance Contract.
- A separate Acceptance pass is required only for risky, user-critical, or explicitly requested work.
- For medium or risky work, planning should usually prepare four low-overlap artifacts:
  - execution task
  - execution evidence report template
  - acceptance contract
  - acceptance review task
- Default acceptance input is `Acceptance Contract + execution evidence + current diff`.
- Any execution task generated by Planning must be clear enough that `执行区` does not need to invent strategy, product behavior, runtime choice, storage design, or other unresolved user decisions.
- Name the four zones in Chinese by default: `规划区`, `执行区`, `验收区`, `维护区`.
- When this skill auto-creates the four zones, use those exact Chinese names by default.
- Auto-bootstrap Git by default with the smallest useful setup: `git init -b main`, a minimal `.gitignore` when missing, and a recorded status in project notes.
- Auto-create the four zone threads by default and send each one its role-specific opening prompt so the user does not have to do it manually.
- When bootstrapping a brand-new workspace, make each new zone's first reply minimal to avoid wasting tokens. Prefer one short acknowledgement or one short status line, not a long analysis.
- Use these default model and thinking settings when creating zone threads:
  - `规划区`: `gpt-5.5` with `high` thinking
  - `执行区`: `gpt-5.5` with `medium` thinking
  - `验收区`: `gpt-5.4` with `medium` thinking
  - `维护区`: `gpt-5.4` with `medium` thinking

## Workflow

1. Read [REFERENCE.md](REFERENCE.md).
2. Copy the templates under `references/`.
3. Customize read order, product docs, commands, and safety rules for the project.
4. If the project lacks authoritative behavior, architecture, or phase docs, seed tracked docs for product consensus, architecture, and implementation phases.
5. Merge with any existing workflow system instead of replacing it wholesale.
6. If Git is missing, initialize it before handing the workspace back to the user.
7. Create four zone threads titled in Chinese, using the zone-specific model and thinking defaults above, and prefill their opening messages.
8. Keep the opening prompts strict about first-reply brevity in empty or newly bootstrapped workspaces.
9. When generating task cards, keep the required read set small and use explicit on-demand read triggers for unchanged docs.
10. Keep the diff minimal; do not add extra zones or files unless the project actually needs them.
